## 0. What Changed Since PRD v1.0 (Research-Driven Updates)

The original PRD (v1.0) was strong on market/SEO but had several material gaps
that 2025-2026 regulatory and ecosystem changes have exposed. v2.0 corrects:

1. **E-invoicing format is no longer OIOUBL.** Denmark officially cancelled
   OIOUBL 3.0 in January 2026 and committed to a single Peppol-based standard
   (Nemhandel BIS 4 / Peppol BIS 3.0 today, migrating to Peppol PINT by 2029).
   v1.0 referenced "OIOUBL export" which is now a dead end.
2. **NemHandel auto-enrollment from 1 July 2026.** All entities on registered
   bookkeeping systems will be enrolled in NemHandel by default and prompted to
   issue e-invoices unless they opt out. This re-positions our "EAN export"
   Business-tier feature: real Peppol connectivity is now table stakes, not a
   nice-to-have.
3. **SAF-T 2.0 export required from 1 January 2027.** Registered bookkeeping
   systems must be able to export SAF-T 2.0 XML. Even though we are a
   "pre-bookkeeping" tool, supporting SAF-T 2.0 export is a strong upsell and
   accountant-handoff feature v1.0 missed entirely.
4. **Lucia Auth is deprecated (March 2025).** v1.0 recommended Lucia Auth.
   Lucia v3 is now a learning resource only; we use **Better Auth** (actively
   maintained, SvelteKit-native, plugins for passkey/2FA/org).
5. **MobilePay is now Vipps MobilePay.** The legacy MobilePay Invoice / POS
   facade APIs were retired; the joint **ePayment API** is the only supported
   path. v1.0's "MobilePay Subscriptions" open question is resolved: integrate
   via Vipps MobilePay ePayment API (or rely on Stripe via Autumn, which wraps it).
6. **Better tooling has landed.** Drizzle ORM on porsager `postgres` is the
   preferred self-hosted-Postgres-on-Workers path; shadcn-svelte is now
   runes-native on Tailwind v4; Autumn (useautumn.com) replaces raw Stripe
   for billing/entitlements with far less webhook code.
7. **Sequential numbering rules clarified.** EU/DK law requires a unique,
   sequential, *unbroken* series, but it may be alphanumeric and may run in
   parallel series. v1.0's "no gaps" rule is stricter than the law requires
   and would block legitimate series-per-year / series-per-currency patterns.
8. **Cookie consent hardened (Datatilsynet May 2025).** New joint guidance with
   DIGST tightened consent requirements; Google Consent Mode v2 is required
   for AdSense to function. Cookiebot/Usercentrics CMP is the de-facto DK
   standard.
9. **PDF strategy must be true-text, not rasterised.** html2canvas+jsPDF
   produces image-only PDFs that fail text-selectability and accessibility and
   are weaker for digital bookkeeping attachments. We standardise on **pdf-lib**
   for true-text PDFs with selectable text and embedded fonts (incl. Danish
   æøå and EUR/DKK currency glyphs).
10. **Self-hosted Postgres + porsager, NOT Neon + Hyperdrive.** Per the
    owner's default stack, we use a flat-rate self-hosted Postgres box
    (e.g. Hetzner CX22 in EU) with PgBouncer (transaction mode) + TLS. The
    **porsager `postgres`** driver connects directly from Workers with
    `ssl:'require', prepared:false`, creating a fresh `postgres()` instance
    per `getDb()` call (`max:1, idle_timeout:5`) — never cached at module
    level on Workers (causes "Cannot perform I/O on behalf of a different
    request" errors). This avoids Neon cold-start costs, Hyperdrive
    complexity, and vendor lock-in.

## 1. Vision & Positioning (unchanged)

Become the default "first invoice" tool for every Danish freelancer and small
business: the lightweight on-ramp that feeds users into real accounting
software (taking affiliate fees along the way).

- NOT a replacement for Dinero / Billy / e-conomic.
- IS the simplest, fastest way to make a compliant Danish invoice.
- IS the bridge between "I just need to send one invoice" and "I need real
  accounting".

## 2. Regulatory Landscape (UPDATED - critical for scope)

### 2.1 Bogføringslov phase-in (complete)

- Class-A companies (>300K DKK turnover two consecutive years): digital
  registered bookkeeping system required since **1 Jan 2026**; in-house
  systems since **1 Jul 2026**.
- ~118,000 additional sole traders + foreign PE entities pulled in from
  1 Jan 2026.

### 2.2 NemHandel default e-invoicing (NEW - effective 1 Jul 2026)

- All entities on registered bookkeeping systems are auto-enrolled in
  NemHandel unless they opt out.
- Registered systems must auto-check NemHandel when a recipient is selected
  and route via Peppol if the recipient is reachable.
- **Implication for us**: A genuinely useful Business-tier feature is
  "Peppol-aware send" (lookup recipient EAS code, offer to deliver via an
  Access Point partner). We do NOT need to become a registered bogføringssystem.

### 2.3 Format transition: OIOUBL -> Peppol (NEW)

- OIOUBL 3.0 cancelled Jan 2026. Existing OIOUBL 2.1 still accepted during
  transition.
- Nemhandel BIS 4 (Peppol BIS 3.0 + DK extensions) is the going-forward
  format.
- Migration to Peppol PINT targeted by 2028-2029.
- EU ViDA directive mandates structured e-invoicing for intra-EU B2B from
  2030; same EN 16931 family as Peppol.

### 2.4 SAF-T 2.0 export (NEW - required from 1 Jan 2027)

- Registered systems must export SAF-T 2.0 XML on demand for Skat / Danmarks
  Statistik / audit.
- We add SAF-T 2.0 XML export as a Pro/Business feature and as the
  accountant-handoff format (replaces ad-hoc CSV for serious users).

### 2.5 Mandatory invoice fields (unchanged from v1.0)

Per Momsloven § 52a + Bekendtgørelse § 58 + EN 16931:
- Fakturanummer (unique, sequential, may be alphanumeric, may be one of
  several parallel series - but each series must be unbroken)
- Udstedelsesdato, Forfaldsdato + betalingsbetingelser
- Sælger: navn, adresse, CVR/SE
- Køber: navn, adresse (EAS code + Peppol ID for e-invoices)
- Leveringsdato if different
- Line items: beskrivelse, mængde, enhed, pris pr. enhed ekskl. moms
- Momsgrundlag, momssats, momsbeløb, total inkl. moms
- Valuta (DKK default)
- For Peppol: BT-24..BT-31 semantic fields per EN 16931

### 2.6 Edge cases (unchanged from v1.0)

| Scenario | Rule |
|---|---|
| Uden CVR (sub-50K) | CPR first 6 digits (fødselsdato); never store full CPR |
| Momsfritaget | "Momsfritaget efter momslovens § 13" |
| EU B2B reverse charge | "Moms omvendt betalingspligt" + buyer VAT |
| Public sector | EAN/GLN + OIOUBL/Peppol via NemHandel |
| Kunstnermoms | 20% moms-pligtig / 80% moms-fri split |
| Proformafaktura | "Proformafaktura - ikke en regning" |
| Kreditnota | Negative amounts, references original |
| Acontofaktura | Moms on each partial payment |

### 2.7 Storage & retention

- Invoices + attachments retained 5 years (Bogføringslov).
- For Pro: encrypted at rest, EU-hosted (self-hosted Postgres EU box +
  Cloudflare R2 EU).
- PDFs must be stored with the structured source data so an accountant or
  auditor can reconcile (relevant for SAF-T 2.0 export).

### 2.8 Liability & GDPR

- Mandatory disclaimer on every page (see v1.0 §6.4).
- Cookie consent: implement via Cookiebot/Usercentrics CMP with Google
  Consent Mode v2 (required for AdSense to fire on content pages).
- CPR: never stored in free tier. In Pro only with explicit consent + DPIA
  on file. Prefer the 6-digit fødselsdato fallback.
- DPO/contact email published; privacy policy + cookie policy pages.

### 4.1 Free Tier (traffic engine)

**Generator:**
- Live PDF preview as user types (debounced, true-text via pdf-lib).
- Auto-calc moms 25% with inkl./ekskl. toggle; kunstnermoms 20/80 preset;
  momsfritaget preset; reverse-charge preset.

## 5. Monetization Strategy (unchanged structure, refined)

### 5.1 Revenue streams (Year 1 / Year 2 / Year 3)

- Pro subscriptions: 2,500 / 16,000 / 35,000 DKK/mo
- Business subscriptions: 1,000 / 5,000 / 15,000 DKK/mo
- One-time purchases: 1,000 / 3,000 / 5,000 DKK/mo
- Affiliate (Dinero / Billy / e-conomic / revisor matchmaking): 2,000 /
  8,000 / 20,000 DKK/mo
- Sponsorships / direct ads: 0 / 2,000 / 5,000 DKK/mo
- AdSense (content pages only): 500 / 3,000 / 6,000 DKK/mo

### 5.2 Payment provider decision

**Primary: Autumn** (`autumn-js` SDK, https://useautumn.com) as the
billing/entitlements layer on top of **Stripe** as the underlying payment
processor. Autumn handles subscription state, one-time purchases, credit
packs, metered usage, feature entitlements, and checkout flows via three
core functions: `attach` (subscribe/checkout), `check` (entitlements
gating), `track` (usage metering). Plans and features are defined in
`autumn.config.ts`. Stripe still processes payments and holds customer
data; Autumn manages all billing logic so we don't need Stripe webhook
handlers for subscription state sync. Stripe still needs to be configured
with the product/price catalogue that Autumn references.

MobilePay is delivered as a Stripe payment method in Denmark where
possible, removing the need for a direct Vipps MobilePay merchant contract.

Products in `autumn.config.ts`:
- `pro` (49 DKK/month, or 470 DKK/year): cloud storage, client DB, SAF-T
  export, rykker emails, cross-device numbering.
- `business` (149 DKK/month): Peppol send, multi-seat org, API access,
  white-label PDFs.
- `lifetime_pro` (999 DKK one-time): all Pro features forever.
- `template_pack` (149 DKK one-time): 10 premium designs.
- `branch_bundle` (249 DKK one-time): niche templates.

### 5.3 Affiliate partners

Dinero (500-1,000 DKK/customer), Billy (500-1,000), e-conomic (1,000-2,000),
revisor matchmaking (200-500/lead), erhvervsansvarsforsikring (100-300/lead),
MobilePay Merchant (revenue share).

### 6.2 Architecture principles

- **SSR/SSG** for SEO-critical pages; CSR for the generator.
- **Client-side PDF generation** with pdf-lib for the free tier (no server
  cost); server-side via Worker for Pro features (email attach, R2 store).
- **EU data residency** end-to-end (self-hosted Postgres EU box, R2 EU,
  Cloudflare EU edge).
- **Progressive enhancement** for crawlers (Googlebot JS-less fallback).
- **LocalStorage-first** for free tier (zero backend cost).
- **No Hyperdrive**: self-hosted Postgres + PgBouncer handles pooling
  server-side; Workers connect directly via porsager with a fresh
  per-request client (`max:1, idle_timeout:5`). Never cache a `postgres()`
  pool at module level on Workers.
- **TypeScript strict** mode; Drizzle schema is the single source of truth.

### 6.3 Data model (Pro tier, Drizzle schema)

Tables: `user`, `session`, `account`, `verification` (Better Auth core) +
`business`, `client`, `invoice`, `invoice_item`, `reminder_log`,
`subscription` (mirrors Autumn customer/product state for Pro/Business).

Key columns on `invoice`:
- `invoice_number` (string, unique per business per series)
- `series` (e.g. `2026`, default per business)
- `status` (draft | sent | paid | overdue | void)
- `currency`, `subtotal`, `vat_rate`, `vat_amount`, `total`
- `items` (jsonb array, also denormalized to `invoice_item` for SAF-T)
- `pdf_r2_key`, `peppol_id` (nullable, Business tier)
- `client_id` (FK)
- `issued_at`, `due_at`, `paid_at`
